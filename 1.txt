# 学习笔记

## 2026-04-22

> 今日主题：RAG 基础、Multi-query、RAG-Fusion、Sub-question、Least-to-most、IR-CoT、Step-back prompting、HyDE


### 知识点整理

#### 1. 基础概念

- 常见的 OpenAI embedding 向量规格里，经常会遇到 `1536` 维这一类表示方式，本质上是把文本映射成高维向量，便于做相似度检索。
- `LangChain` 里很多对象都可以通过 `.invoke()` 做同步调用，常见对象包括 `model`、`chain`、`tool`、`prompt`。
- 当原始问题过于模糊、一次检索不容易命中时，可以先引入 `Multi-query`，把问题改写成多个问法，提高召回率。

#### 2. Multi-query

- `Multi-query` 的核心思路：先让 LLM 把一个原始问题改写成多个相关 query，再让每个 query 分别去检索。
- 它的重点是“多问法召回”，也就是尽量扩大命中范围，避免因为用户提问方式单一而漏掉相关文档。
- 每个子 query 检索出来的结果通常都有自己的局部排名，因为检索器本身就会返回一个 `top-k` 列表。
- 但 `Multi-query` 本身不一定天然给出“所有结果融合后的统一全局排名”。

#### 3. RAG-Fusion

- `RAG-Fusion` 可以看作 `Multi-query` 的增强版。
- 它不仅会生成多个 query 去分别检索，还会把多个结果列表做进一步融合。
- 它的重点不只是“多问法召回”，还包括“多结果融合排序”。
- 更准确地说，`RAG-Fusion` 会综合一个文档在多个 query 检索结果中的表现，生成一个新的全局 rank。

#### 4. RAG-Fusion 里的 rank 是怎么来的

- `RAG-Fusion` 的 rank 不一定是 LLM 逐篇理解文档后重新做语义评分。
- 很多实现里，它更像是对多个 query 检索结果中的名次做融合。
- 如果某篇文档只在一个 query 下排得高，它可能只是局部相关。
- 如果某篇文档在多个 query 下都排得比较靠前，它通常会在融合后得到更好的全局排名。
- 所以融合后的 rank，本质上更像“多次检索证据汇总后的排序”。

#### 5. LLM 会不会参考 rank

- 会受影响，但不会机械地只认 rank。
- 如果系统只是按 rank 顺序把文档喂给 LLM，那么排在前面的文档会更容易先被模型看到。
- 如果系统把 `rank` 或 `score` 显式写进 prompt，LLM 也可能把它当成参考信号。
- 但最终回答仍然取决于 LLM 如何综合理解这些文档内容，而不是简单执行“第一名一定正确”。

#### 6. 为什么不直接只取最高 rank 的文档

- 第一名文档不一定完整。
- 第一名文档也不一定绝对正确。
- 复杂问题往往需要多个文档互相补充，才能形成更完整的答案。
- 如果只依赖一篇文档，一旦这篇文档检索偏了，最后答案也容易整体偏掉。
- 所以实际系统通常会保留完整排序，再从中选 `top-k` 交给 reranker 或 LLM。

#### 7. Sub-question 是什么

- `Sub-question` 的核心是“问题分解”，不是“问题改写”。
- 它会把一个复杂问题拆成几个更小、信息需求不同的子问题，再分别检索或回答。
- 这种方法常用于多跳问答、复杂比较题、需要多步推理的问题。
- 如果原问题本身包含多个隐含条件或多个推理环节，`Sub-question` 往往比单纯 `Multi-query` 更有效。

图示里这种结构，属于并行式 `Sub-question`，也可以理解为 `Decomposition RAG`：

- 先把一个原问题拆成多个子问题。
- 每个子问题分别检索各自需要的文档。
- 每个子问题分别生成局部答案。
- 最后再把多个局部答案聚合成最终答案。

这类结构的重点是“先拆分，再并行处理，最后汇总”，因此它属于 `Sub-question`，但不属于 `Least-to-most` 或 `IR-CoT`。

和 `Multi-query` 的关键区别：

- `Multi-query` 更像是“同一个意思，换几种问法去检索”。
- `Sub-question` 更像是“把一个大问题拆成几个不同的小问题分别解决”。

#### 8. Least-to-most

- `Least-to-most` 是一种分解式推理方法。
- 它的思路是：先解决最简单、最基础的子问题，再利用前面得到的答案，逐步解决更难的子问题，最后完成原问题。
- 它的重点不是一次把所有子问题并行列出来，而是强调“从简单到复杂、逐步推进”。
- 如果说 `Sub-question` 只是“拆分”，那么 `Least-to-most` 更像是“按依赖关系分步骤求解”。

它特别适合有明显推理链条的问题，例如：

- 先确认定义或前提。
- 再判断中间条件。
- 最后给出综合结论。

#### 9. IR-CoT

- `IR-CoT` 可以理解为“检索和推理交替进行”。
- 它和 `Sub-question` 有重叠，但不严格属于“先拆好子问题再逐个解决”的那一类。
- 更准确地说，`IR-CoT` 的重点不是“预先拆分”，而是“边推理边发现信息缺口，再继续检索”。
- 它不是先一次性把所有资料都取回来再统一思考，而是推理到某一步时，发现缺信息，就继续检索下一步需要的证据。
- 所以 `IR-CoT` 特别适合信息需求会随着推理过程不断变化的问题。
- 和普通 RAG 相比，它不是“先检索、后生成”这么简单，而是“检索 <-> 推理”交替迭代。

它的核心流程更像：

- 先读原问题。
- 先做一小步推理。
- 判断当前缺什么信息。
- 把这个信息缺口变成一次针对性检索。
- 再结合新证据继续下一步推理。
- 如果还缺信息，就继续进入下一轮“推理 -> 检索”。
- 最后再汇总得到答案。

和 `Sub-question` 的流程区别：

- `Sub-question` 更像“先拆后解”。
- `IR-CoT` 更像“边解边拆，边拆边检索”。
- 因此，`IR-CoT` 在执行过程中可能会产生临时子问题，但通常不是一开始就把所有子问题完整列出来。

#### 10. Sub-question、Least-to-most、IR-CoT 三者关系

- `Sub-question` 是总思路，重点是把复杂问题拆开。
- `Least-to-most` 是 `Sub-question` 的一种更强调顺序依赖的做法。
- `IR-CoT` 则更强调在推理过程中动态检索，不一定一开始就把所有子问题一次性列完，也不严格属于预先规划型的 `Sub-question`。

可以粗略理解成：

- `Sub-question`：先拆问题。
- `Least-to-most`：按从易到难的顺序解决拆出来的问题。
- `IR-CoT`：边想边查，查完再继续想。

#### 11. 这些方法最容易出错的地方

- 把 `Multi-query` 和 `Sub-question` 混为一谈。
- 前者是“同义改写”，后者是“任务拆解”，两者目标不同。
- 子问题拆得过碎，导致检索次数过多、成本上升、噪声变大。
- 子问题拆得不对，后面即使每步都执行正确，最终答案也会被错误分解带偏。
- 在 `Least-to-most` 里，如果前一个简单问题答错了，后面的步骤会层层继承这个错误。
- 在 `IR-CoT` 里，如果某一步检索方向跑偏，后续推理会围绕错误证据持续展开。
- 只关注拆分步骤，却忽略最后的答案整合，导致每个子问题都答了，但最终结论没有真正收束。
- 把检索到的文本直接当成绝对事实，而没有做交叉验证或上下文核对。

#### 12. Step-back prompting 是什么

- `Step-back prompting` 的核心不是把问题拆开，也不是把问题换几种说法，而是先“后退一步”，把具体问题提升成一个更抽象、更一般化的问题。
- 这样做的目的是先拿到更高层的背景知识、通用原理或上位概念，再回到原问题作答。

在检索增强场景里，它常见的流程是：

- 先读取原始问题。
- 让 LLM 生成一个更抽象的“上位问题”。
- 用这个上位问题去检索背景知识或原则性文档。
- 再把这些抽象知识和原始问题一起交给模型，生成最终答案。

可以把它理解成：

- 原问题在问“某个具体点”。
- `Step-back prompting` 先去问“这个具体点背后的通用规律是什么”。

#### 13. Step-back prompting 的核心特点

- 它强调“抽象化”，不是“分解化”。
- 它特别适合原问题太具体、太贴边、太依赖背景原则时的场景。
- 当直接检索原问题只能命中零散片段，但更高层的概念、定义、原则其实很重要时，它很有用。
- 它常常能补足“具体问题检索不到，但相关原理文档很多”的情况。
- 它也可以和普通检索结合：既检索原问题，也检索 step-back 后的抽象问题，再一起交给 LLM。

#### 14. Step-back prompting 和前面几种方法的区别

和前面方法相比，它最大的不同是：

- `Multi-query`：同一个问题换几种问法，提高召回率。
- `RAG-Fusion`：把多路检索结果融合，得到更稳的全局排序。
- `Sub-question`：把复杂问题拆成多个不同子任务。
- `Least-to-most`：把子任务按依赖顺序，从简单做到复杂。
- `IR-CoT`：边推理边发现缺口，再继续检索。
- `Step-back prompting`：先把问题上提一个层级，先找通用原则，再回到具体问题。

所以它和其他方法的差异可以一句话记成：

- 前面几种大多是在“怎么拆、怎么查、怎么排”上做增强。
- `Step-back prompting` 更像是在“先看更高层的知识框架，再回来回答细节”。

#### 15. Step-back prompting 最容易错的地方

- 抽象得过头，导致检索结果太泛，最后拿回来一堆正确但无用的背景知识。
- 只拿抽象问题做检索，却忘了回到原始具体问题，导致答案变空泛。
- 把它误当成 `Multi-query`，其实它不是同义改写，而是层级上移。
- 把它误当成 `Sub-question`，其实它不是拆多个任务，而是把同一个任务上提到更一般的层次。
- 当问题本身已经很抽象时，再做 `Step-back prompting` 往往收益不大，反而可能让检索更偏。

#### 16. 这些方法分别适合什么场景

- `Multi-query`：适合原问题表述可能单一、容易漏召回，但核心任务其实没变的场景。
- `RAG-Fusion`：适合已经有多路检索结果，想把它们融合成更稳排序的场景。
- `Sub-question`：适合一个问题本身包含多个信息需求、多个条件或多个子任务的场景。
- `Least-to-most`：适合子问题之间有明显依赖关系，必须一步接一步解决的场景。
- `IR-CoT`：适合信息缺口会随着推理过程动态变化的场景，尤其是多跳问答。
- `Step-back prompting`：适合原问题太具体、太贴边，直接检索不到好材料，但其背后存在较清晰的通用原理、背景知识或上位概念的场景。

#### 17. 该如何快速辨别该用哪一种

- 如果你觉得“这个问题其实没变，只是换个问法也许更容易检索到”，优先想到 `Multi-query`。
- 如果你觉得“我已经查到了很多候选结果，但排序不稳定”，优先想到 `RAG-Fusion`。
- 如果你觉得“这个问题本身包含几个不同的小任务”，优先想到 `Sub-question`。
- 如果你觉得“这些小任务有前后依赖，后一步要用前一步结果”，优先想到 `Least-to-most`。
- 如果你觉得“我只有推到一半才知道下一步该查什么”，优先想到 `IR-CoT`。
- 如果你觉得“我不是缺具体答案，而是先缺一个更高层的原则或背景框架”，优先想到 `Step-back prompting`。
- 如果你觉得“用户问题太短、太模糊，和真实文档写法差得很远，直接拿 query 去向量检索不容易命中”，优先想到 `HyDE`。

一个很实用的辨别口诀：

- 换问法：`Multi-query`
- 融结果：`RAG-Fusion`
- 拆任务：`Sub-question`
- 按顺序拆：`Least-to-most`
- 边想边查：`IR-CoT`
- 先抬高一层再回答：`Step-back prompting`
- 先生成一篇“像答案的假文档”再去检索：`HyDE`

#### 18. HyDE 是什么

- `HyDE` 是 `Hypothetical Document Embeddings` 的缩写。
- 它的核心不是直接把用户问题拿去做向量检索，而是先让 LLM 根据问题“写出一篇假想的、看起来像正确答案的短文档”，再对这篇假文档做 embedding，用它去检索真实文档。
- 重点在于：真正进入向量检索阶段的，不再只是原始 query，而是“由 LLM 生成的假想文档的向量表示”。
- 最后系统返回的仍然应该是真实检索到的文档，而不是把那篇假想文档直接当答案。

可以把它理解成：

- 用户的问题往往很短。
- 真实文档往往写得更完整、更像一段说明文字。
- `HyDE` 先把“短问题”扩成“一段像文档的文本”，再去和真实文档做向量匹配。

#### 19. HyDE 的核心知识点

- 它的目标是缓解“query 和 document 表达形态不一致”的问题。
- 普通检索里，用户问题可能只有一句话，但知识库文档通常是更长、更完整、更带上下文的段落。
- `HyDE` 通过先生成一篇假想文档，让检索输入更接近真实文档的写法，从而提升 dense retrieval 的命中率。
- 它本质上仍然是“为了更好地检索真实文档”，不是让模型直接幻想答案后停止。
- 所以 `HyDE` 的关键价值，不在假文档内容本身，而在“这篇假文档的 embedding 是否更接近真实相关文档”。

#### 20. HyDE 和前面几种方法的区别

- `Multi-query` 是把同一个问题换几种问法。
- `Step-back prompting` 是先把问题提升到更抽象的层级。
- `Sub-question` 是把一个复杂问题拆成多个子任务。
- `IR-CoT` 是边推理边检索。
- `HyDE` 则是先生成一篇“像答案文档”的假文本，再拿这篇文本的向量去找真实文档。

所以一句话区分：

- `Multi-query` 解决的是“问法不够多”。
- `Step-back prompting` 解决的是“缺上位背景知识”。
- `HyDE` 解决的是“问题写法和文档写法差太远”。

#### 21. HyDE 的使用注意事项

- 不要把 LLM 生成的假想文档直接当成最终答案，它只是检索用的中间桥梁。
- 如果假想文档 hallucination 很严重，它可能会把检索方向带偏，导致召回到一批“围绕错误细节展开”的文档。
- 对高风险场景，最好把 `HyDE` 检索结果和原始 query 检索结果结合，再做 rerank 或交叉验证。
- 当原问题已经写得很完整、很像文档描述时，`HyDE` 的增益可能不明显。
- 如果知识库内容特别短、特别结构化，`HyDE` 也不一定比普通检索更有优势。
- 比较稳妥的做法通常是：保留原 query 检索作为基线，再把 `HyDE` 作为增强通道，而不是完全替代原始检索。

#### 22. HyDE 适用场景

- 用户问题很短、信息稀疏、描述不完整，但知识库文档写得比较长、比较像解释性段落。
- 查询和文档之间存在明显“表达鸿沟”，也就是用户说法和资料写法差异很大。
- 零样本或弱监督检索场景，没有太多人工标注数据去训练更强的检索器。
- 专业领域检索里，用户可能只会问一个现象或症状，而文档是成体系的技术说明、论文段落或产品文档。
- 你已经感觉普通向量检索“不是完全没结果，而是结果总有点擦边”，这时 `HyDE` 往往值得尝试。

### 今日结论

- `Multi-query` 的重点是把一个问题改写成多个 query，提高召回率。
- 每个子 query 的检索结果通常都有自己的局部排名。
- `RAG-Fusion` 是在 `Multi-query` 基础上再做结果融合，得到新的全局 rank。
- rank 会影响 LLM 的输入组织方式，但不会直接决定最终答案。
- 返回完整排序不是多余，而是为了给后续筛选、重排、生成和调试留下空间。
- `Sub-question` 的重点不是改写，而是把复杂问题拆成不同的信息需求。
- `Least-to-most` 强调按从简单到复杂的顺序逐步求解。
- `IR-CoT` 强调推理和检索交替进行，适合多跳、动态信息需求的问题。
- `Step-back prompting` 强调先抽象、先看上位原则，再回到具体问题作答。
- `HyDE` 强调先生成一篇假想文档，再用这篇文档的 embedding 去检索真实资料。
- 这几种方法并不是互斥的，很多系统会把它们组合使用。
- 判断该用哪一种，关键看你当前缺的是：更多问法、统一排序、任务拆分、顺序求解、动态检索、高层背景知识，还是 query 和 document 之间的表达桥梁。

### 今日易混点

- `Multi-query` 不等于 `RAG-Fusion`。
- 子 query 下的 rank 是局部 rank，融合后的 rank 才是全局 rank。
- 检索 rank 决定“优先看哪些文档”，但不等于“LLM 只能照着第一名回答”。
- 不直接取最高 rank 输出，是因为真实问答往往需要多个候选文档共同支撑。
- `Multi-query` 不等于 `Sub-question`。前者是改写同一问题，后者是拆分不同子任务。
- `Least-to-most` 不只是把问题拆开，而是要求前后步骤存在依赖关系。
- `IR-CoT` 不等于普通 RAG。普通 RAG 常是先检索后生成，`IR-CoT` 更像边检索边推理。
- `Step-back prompting` 不等于 `Multi-query`。它不是换说法，而是把问题提升到更抽象的层次。
- `Step-back prompting` 不等于 `Sub-question`。它不是拆成多个任务，而是先补上位知识框架。
- `HyDE` 不等于 `Multi-query`。它不是多改写几个 query，而是先生成一篇像答案的假文档。
- `HyDE` 不等于 `Step-back prompting`。前者是在表达形态上把 query 变得更像文档，后者是在知识层级上把问题抬高。

## 2026-04-24
> 今日主题：逻辑路由和语义路由

### 知识点整理

#### 1. 路由在 RAG 或 LLM 系统里是什么意思

- 这里的“路由”可以理解为：系统先判断“这个问题应该走哪条处理路径”，再决定后续调用哪个检索器、知识库、工具、工作流或模型。
- 路由的核心目的不是直接回答问题，而是先做“分流”。
- 在复杂系统里，路由常常决定：
- 这个问题该去查哪个知识库。
- 该走普通问答、检索增强，还是工具调用。
- 该交给便宜模型还是更强模型。
- 该用结构化规则处理，还是交给语义理解模块处理。

#### 2. 逻辑路由是什么

- 在你当前这套语境里，`逻辑路由` 更准确地说是“由大模型根据问题内容、任务目标和各条路由的说明，推理后决定应该走哪条路径”。
- 它不是单纯看某个关键词有没有出现，而是让模型判断：这个请求从任务逻辑上更应该被哪个处理链路接管。
- 它通常会把候选路由的职责描述、适用范围、输入要求一起交给模型，再让模型做一次判别。
- 所以这里的“逻辑”，更接近“基于任务理解做判断”，而不是“硬编码 if-else 规则”。

可以把它理解成：

- 让 LLM 读完问题后判断：“这个请求按任务逻辑更应该去哪处理？”

例如：

- 用户问“我这个问题应该去查公司制度还是走工单处理”，模型根据问题真实意图判断应该走制度问答还是流程执行链路。
- 用户问“帮我看看这个需求是该查知识库还是该调用工具”，模型根据任务性质判断是检索型请求还是操作型请求。
- 用户问法很自由，但模型仍能根据各路由的职责说明，选出更符合任务逻辑的链路。

#### 3. 语义路由是什么

- `语义路由` 的核心是“把用户输入和各个路由的语义表示做匹配，看它在向量空间里更接近哪一类”。
- 它通常依赖 embedding 相似度、语义分类器或向量检索来决定路由归属。
- 和逻辑路由不同，语义路由更像是在做“语义最近邻匹配”，而不是让模型显式地做一步任务推理判断。

可以把它理解成：

- 不是问“模型觉得从任务逻辑上该去哪”。
- 而是问“这个输入和哪类路由描述在语义上最像”。

例如：

- 用户问“假期审批一般提前多久提”，它和 HR 制度类路由的描述在语义上最接近，就可能被分过去。
- 用户问“客户退款一直没到账怎么办”，它和售后流程、订单异常处理相关的路由向量更近，就可能被路由到对应链路。

#### 4. 逻辑路由和语义路由的核心区别

- `逻辑路由` 更强调“由模型理解任务后做判断”。
- `语义路由` 更强调“输入和路由描述在语义空间里谁更接近”。
- `逻辑路由` 往往会显式利用每条路由的职责、边界、使用条件，让模型做带解释色彩的选择。
- `语义路由` 往往更像相似度匹配，速度更快，也更容易工程化批量处理。
- `逻辑路由` 通常更适合处理边界复杂、需要综合判断的问题。
- `语义路由` 通常更适合主题区分明确、可以靠语义接近度完成分流的问题。

一句话区分：

- `逻辑路由`：让模型判断“从任务逻辑上该去哪”。
- `语义路由`：看输入“和哪条路由在语义上最像”。

#### 5. 逻辑路由的优点和局限

优点：

- 能综合考虑任务目标、上下文、路由职责和边界条件。
- 对复杂请求、更隐含的用户意图通常理解得更好。
- 当多个路由语义上都相近，但职责不同的时候，逻辑路由更容易分清。
- 可以要求模型顺手给出“为什么选这条路由”的理由，可解释性不一定差。

局限：

- 成本通常更高，因为往往要调用 LLM 做判断。
- 时延通常高于纯向量相似度匹配。
- 如果提示词设计不好，模型可能会在边界问题上摇摆。
- 对输出格式、路由标签约束不严时，可能出现不稳定选择。

#### 6. 语义路由的优点和局限

优点：

- 推理成本通常比逻辑路由低，更容易做成高吞吐分发层。
- 工程实现简单，常见做法是把路由说明做 embedding 后和用户输入比相似度。
- 对同义表达、口语化表达、简短 query 也有一定鲁棒性。
- 当各路由主题边界比较清晰时，效果通常不错。

局限：

- 它更擅长“像不像”，不一定擅长“该不该”。
- 当多个候选路由语义相近但职责边界不同，语义路由更容易混淆。
- 它不一定真正理解任务约束、业务前提和处理顺序。
- 可解释性通常弱于让 LLM 显式给出判断理由的逻辑路由。

#### 7. 各自适用场景

`逻辑路由` 更适合：

- 多个路由职责接近、仅靠语义相似度不容易分开的场景。
- 需要结合任务目标、权限、步骤要求做综合判断的场景。
- 希望让模型根据路由说明进行更细粒度裁决的场景。
- 复杂 Agent 系统里的任务分派。

`语义路由` 更适合：

- 路由主题区分度较高的问答系统。
- 希望低成本、低时延做第一层分流的场景。
- 企业知识库、客服、FAQ 这类主题簇相对清晰的系统。
- 适合先粗分流，再把结果交给后续更强判断模块。

#### 8. 实际系统里常见的做法

- 很多成熟系统并不会只选一种，而是“逻辑路由 + 语义路由”组合使用。
- 常见方式是先用语义路由做粗分类，再用逻辑路由做精判。
- 也可能先用硬规则做第一层拦截，再把剩余请求交给语义路由或逻辑路由。

一个常见组合例子：

- 先用 embedding 看这个问题更像 HR、财务还是售后。
- 再把候选的 2 到 3 条路由说明交给 LLM，让它根据任务逻辑最终决定走哪一条。

### 今日结论

- 路由的本质是先决定“这个问题该走哪条处理路径”。
- 在你这套语境里，`逻辑路由` 更接近“由大模型做任务级判断”。
- `语义路由` 更接近“基于 embedding 或语义相似度做匹配分流”。
- `逻辑路由` 更擅长复杂边界判断，`语义路由` 更擅长低成本快速分流。
- 真正的系统里，两者通常不是二选一，而是组合使用。
- 很多系统会先用语义路由粗筛，再用逻辑路由精判，必要时再加规则兜底。
