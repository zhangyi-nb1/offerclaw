# OfferClaw · 伦理使用边界（Ethical Use）

> 用途：明确 OfferClaw 哪些事情**不做**，以及为什么不做。  
> 这不是法律文书，是**项目自律契约**。任何代码 PR 违反本文档原则的，应被驳回。

---

## 0. 总则

OfferClaw 是辅助求职者**思考、规划、记录**的 AI 助手，不是替求职者投递、伪造、谈判的自动机。

> **OfferClaw 帮你跑得更稳，不替你跑。最终的简历和投递决定永远由用户做出。**

---

## 1. 红线（绝对不做）

### 1.1 不自动投递
- 不调用 Boss / 拉勾 / LinkedIn / Greenhouse / Lever 等任何招聘平台的提交接口
- 不模拟键鼠操作 / 浏览器自动化去点"投递"按钮
- 即使用户明确请求"帮我一键投 100 个"，也拒绝并解释为什么
- **原因**：自动投递违反招聘平台 ToS，且容易把用户和招聘方双方都推到合规灰区。

### 1.2 不伪造经历
- 不在简历上添加用户没做过的项目、奖项、绩点、实习
- 不"改写"用户的经历到失真程度（"参与"→"主导"是边界，需用户确认）
- 简历每一句话必须有 user_profile.md 中可查证的事实背书
- **原因**：伪造一旦被背调发现，撤 Offer + 行业拉黑。

### 1.3 不承诺录用概率
- 不输出"你有 80% 概率拿到 Offer"这类伪量化结论
- 三档结论（适合 / 暂不 / 中长期可转向）只表达**匹配度**，不表达**录用率**
- 任何"必拿 Offer / 保过 / 包面试"措辞都视为失败回答

### 1.4 不批量骚扰招聘方
- 不批量给 HR 发邮件 / 加微信 / 打电话
- 不冒充猎头 / 内推人 / 校友
- 单个岗位的跟进次数有上限（建议 ≤ 2 次，间隔 ≥ 7 天）

### 1.5 不绕过招聘平台规则
- 不教用户用 VPN / 假身份注册多账号
- 不爬取需要登录或付费的 JD 全文
- 公开 JD 摘录后，必须保留来源链接和抓取时间

### 1.6 不批量抓取 / 不门户扫描（JD Discovery 边界）
- `job_discovery.py` 仅做**单 URL 半自动抽取**，每次只处理用户提供的一个 URL
- **不**实现 Greenhouse / Ashby / Lever / Boss / 拉勾 等招聘门户的批量遍历
- **不**登录任何招聘平台、**不**绕过验证码、**不**模拟用户长时间停留
- Playwright 仅作 SPA 渲染兜底，单次访问，不做翻页 / 列表抓取
- 所有抽取结果带 `source_type` + `source_credibility` 字段（A=公司官网 / B=招聘平台 / C=未知），让用户自己评估可信度
- 任何写入 `applications.md` 的动作都必须用户在 UI 二次确认，**Agent 不静默落库**
- **原因**：批量抓取等同爬虫，违反平台 ToS；门户扫描会把 OfferClaw 从"求职助手"拉进"招聘爬虫工具"叙事，偏离 AI 应用 / Agent 工程的求职定位。

## 2. 信息源与可信度

与 `source_policy.md` 对齐：

| 等级 | 来源 | OfferClaw 行为 |
|---|---|---|
| **A** | 公司官网 / 政府文件 | 可直接进入 user_profile / applications |
| **B** | 知名媒体 / LinkedIn 官方 | 进入 jd_candidates 摘录段，必须标注来源 |
| **C** | 论坛 / 匿名爆料 / 截图 | 仅作参考，不写入用户层；面试相关传言一律标注"未经证实" |

**永不**把 C 级当 A 级使用。

---

## 3. 用户数据保护

- API Key、邮箱、电话、身份证号 → `.env.local` / `.gitignore`，**绝对**不入 Git
- 真实简历 PDF / 内推码 / HR 内部联系方式 → 不入仓库
- 多人使用时（V2 计划）必须按 user_id 隔离 memory.json / profile.md

---

## 4. 用户最终确认（Human-in-the-loop）

以下操作 OfferClaw 只能**生成草稿**，不能**直接执行**或**直接合并**：

| 操作 | 草稿位置 | 合并方式 |
|---|---|---|
| 修改 user_profile.md 中"姓名/学校/绩点" | Agent 输出建议 | 用户手动 edit |
| 在 applications.md 把状态改为"已投递"/"面试中"/"已 Offer" | Agent 提示"建议改为 X" | 用户手动改 |
| 写入 interview_story_bank.md 新故事 | Agent 输出 STAR 草稿 | 用户审 + 改 |
| 给招聘方写邮件 / 私信 | Agent 输出文案 | 用户复制粘贴自发 |

---

## 5. 模型与输出

- **不**为模型输出做"绝对正确"背书；任何 LLM 答案末尾默认隐含"本回答可能有误，请验证"
- 关键决策（投/不投、改方向、退学换专业）一律给出多个选项 + 利弊，不给单一推荐
- 在面试故事 / 简历改写场景，输出末尾标注"草稿"，提醒用户确认

---

## 6. 数据保留与删除

- 用户随时可以删除 `memory.json` / `daily_log.md` / `applications.md`
- 删除后 OfferClaw **不**从 git 历史 / cloud 备份"恢复"——尊重用户的"被遗忘权"
- `chroma_db/` 可以随时 `rm -rf` 重建，不存敏感原文（仅向量 + chunk_id）

---

## 7. 失败模式声明

OfferClaw 当前**已知**的失败模式（V1 阶段）：
1. 长尾岗位（< 50 份训练 JD 见过的方向）匹配可能误判
2. RAG 在跨文档证据组合问题上召回不足
3. LLM 偶发幻觉，特别是在公司财报 / 薪资数据这类强事实领域
4. 没有自动核实 JD 是否仍在招（可能给出已过期 JD 的匹配结论）

**不**对上述失败模式造成的后果负责；用户需自行二次验证关键事实。
