# OfferClaw · 目标与匹配规则（Target Rules）

> 本文件定义 OfferClaw 在做求职方向判断与岗位匹配分析时必须遵守的规则。
> 规则的目的是：**可解释、可追溯、可收敛**。禁止玄学综合打分。

---

## 1. 求职方向优先级

当前锁定的候选方向（按优先级）：
1. Agent 应用工程
2. AI 应用开发
3. Prompt / Workflow 工程

派生方向（可接受，但不优先）：
- LLM 应用后端（以 Python 为主）
- 数据 / 算法落地岗（与 LLM 场景相关）

明确不考虑：
- 以 Java 为主线的业务后端
- 纯前端
- 纯硬件 / 嵌入式

---

## 2. 岗位分析的输入

任何匹配分析都必须显式使用以下输入：
- `user_profile.md` 中的字段（在输出里必须**指名引用章节**）
- 用户当次提供的 JD 文本或链接（JD 由用户主动粘贴，OfferClaw 不自行爬取）
- 若 JD 信息不全，必须先列出缺失项，**不允许脑补补齐**

---

## 3. 硬性门槛检查（Hard Gate）

按顺序检查下列条目，任一未命中都会直接影响结论：

| 门槛项 | 判断方式 |
|---|---|
| 学历要求 | JD 要求 vs 用户学历（profile §1） |
| 专业要求 | JD 所列专业范围 vs 用户专业（profile §1） |
| 经验年限 | JD 要求 vs 用户实习/项目年限（profile §4/§7） |
| 语言要求 | 是否要求英语 / 其他语种（profile §9） |
| 地域要求 | 是否强制线下及具体城市（profile §1） |
| 技术主线 | 是否强制 Java / C++ 等非优先主线（profile §2） |

每一项必须输出："命中 / 未命中 / 信息不足 + 简短理由"。

---

## 4. 软性维度分析（Soft Dimensions）

只做"命中 / 部分命中 / 未命中"的判断，**不输出加权总分**。

| 维度 | 判断依据 |
|---|---|
| 技能重叠 | JD 要求技能 vs profile §3 技能清单 |
| 方向一致度 | JD 所属方向 vs profile §2 求职方向 |
| 项目 / 经历契合 | JD 偏好经验 vs profile §4 / §7 |
| 工作性质匹配 | 远程 / 混合 / 驻场 vs profile §2 |
| 薪资匹配 | JD 区间 vs profile §2 期望薪资 |
| 地域匹配 | JD 所在地 vs profile §1 可接受地域 |

---

## 5. 三档结论（只能输出其一）

- **当前适合投递**：硬门槛全部命中或仅有 1 项"信息不足"；软性维度多数命中。
- **当前暂不建议投递**：存在至少 1 项硬门槛未命中；或软性维度大面积未命中。
- **中长期可转向**：硬门槛基本命中但软性维度差距较大，属于值得规划、不宜立即投递。

---

## 6. 每次输出必须包含的结构

```
岗位：<JD 标题>
方向判定：<主方向 / 派生方向 / 不考虑>

硬门槛：
  - 学历：✓ / ✗ / ?（理由）
  - 专业：...
  - 经验：...
  - 语言：...
  - 地域：...
  - 技术主线：...

软性维度：
  - 技能重叠：命中 / 部分命中 / 未命中（理由 + 引用 profile §x）
  - 方向一致度：...
  - 项目契合：...
  - 工作性质：...
  - 薪资：...
  - 地域：...

结论：<当前适合投递 / 当前暂不建议投递 / 中长期可转向>

缺口清单（喂给路线规划模块）：
  - 硬门槛缺口：<...>
  - 技能缺口：<...>
  - 经历缺口：<...>

下一步建议：<最多 3 条可执行项>
```

---

## 7. 缺口清单的用途

缺口清单是本规则文件与"路线规划"模块之间的**唯一接口**。
- 规划模块只读缺口清单，不再读原始 JD
- 每条缺口必须可以转成一条具体学习/练手任务，否则视为无效缺口

---

## 8. 禁止条款

- 禁止输出 0–100 的综合打分
- 禁止只写结论不写硬门槛与缺口
- 禁止引用非 A/B 级来源作为硬门槛判断依据（见 `source_policy.md`）
- 禁止承诺薪资、录用概率、面试通过率
- 禁止对具体公司做主观评价
